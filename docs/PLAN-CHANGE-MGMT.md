# IntraWorks — 변경요청·품질문서·인증 통합 관리 플랜

> **작성일**: 2026-04-30
> **작성 배경**: SW팀이 현재 폴더(`03.SW/01.제품/C.AIXAC-RX6040DA/`) 단위로 기능 추가·기능 개선·버그 수정마다 폴더를 만들고 그 안에 품질문서(SW성능시험·품질검사 결과)를 보관 중. KTL·항공보안인증 등 외부 검사 시 "기능 → 시험 → 버전 → 배포 → 인증"의 이력 추적이 핵심. 이걸 IntraWorks에 통합 관리하면 좋을지 검토.
> **이 문서는 플랜만 — 구현 X**

## 1. 현 상태 정리

### 1.1 사용자가 이미 운영 중인 흐름

```
03.SW/01.제품/C.AIXAC-RX6040DA/
├── 00.SW시험                          # 모델 단위 시험 보관소
├── 01.기능추가_Detector              # ← 변경요청(CR) 1건 = 폴더 1개
├── 02.기능추가_PLC                    #   안에 시험계획서, 시험결과서,
├── 03.기능추가_UI                     #   품질검사결과, 승인서 등
├── 07.버그수정_PLC 인터락 오류 수정
├── 08.기능개선_물질분별 알고리즘 개선
├── 14.UI_시스템 디자인 리뉴얼
└── 99.old
```

→ **각 번호 폴더 = 1개의 "변경요청(Change Request, CR)"**
→ 폴더명 패턴: `{NN}.{유형}_{설명}`
→ 폴더 안에는 워드/한글/엑셀/PDF 품질문서 다수

### 1.2 버전 규칙 (이미 정해진 것)

`AIXAC.v.[Major].[Minor].[Revision].[Build]`

| 자리 | 변경 시점 |
|------|----------|
| **Major** | 아키텍처 교체, 하드웨어 사양 근본 변화 |
| **Minor** | 신기능 추가, UI/UX 유의미한 변화 |
| **Revision** | 로직 개선, 알고리즘 최적화, 마이너 UI |
| **Build/Patch** | 긴급 수정, 현장 대응 |

→ **CR 한 건 또는 여러 건이 모여 → 버전업 1회**

### 1.3 배포 이력 시트 (Google Sheets)

| 컬럼 | 예시 |
|------|------|
| 프로젝트명 | AIXAC.RX.6040DA |
| 버전번호 | AIXAC.D.v.2.0.0.0 |
| 업데이트 유형 | UI/UIX |
| 변경 사유 및 내용 | 전체적인 UI/UX 개편 |
| 담당자 | 고형균 |
| 배포 날짜 | 2026-02-23 |

→ **현 IntraWorks의 `Release` + `DeployHistory`로 이미 커버됨**.
→ **CR ↔ 릴리스 매칭이 빠져 있음** (이게 이번 플랜의 핵심).

### 1.4 인증/사이트별 시트

탭 구성: `우정국 / 구치소 / 필리핀 / 싱가폴 / 대만 / 6040DA(KTL) / 6040SA(KTL)`

→ 사이트별 또는 모델·인증기관별 배포 이력을 별도 관리 중
→ "이 모델의 KTL 인증 시점 버전이 뭐였지?" 형태 질문 빈번

## 2. 핵심 갭 분석

| 영역 | 현재 (폴더+엑셀) | 갭 |
|------|------------------|-----|
| 변경요청(CR) 생성 | 폴더 만들고 번호 붙임 | 검색 불가, 모델·기간·유형 필터 안 됨, 진행 상태 모름 |
| 품질문서 보관 | 폴더 안 파일들 | 어느 CR과 어느 버전·시험과 연결되는지 메타데이터 없음 |
| 버전 ↔ CR 매핑 | 사람이 기억 / 변경 사유 자유 텍스트 | KTL 검사 시 "v2.1.0.0에 어떤 CR들이 들어있나?" 즉답 불가 |
| 시험 결과 추적 | 폴더 안 보고서 PDF | "PLC 통신 개선" CR의 최종 시험 통과 여부, 수행자, 일자가 표면화 안 됨 |
| 인증 이력 | 시트 탭 분리 | 인증 만료일·갱신·차기 인증 예정 알림 없음 |
| 검사 대응 | 폴더 뒤져서 수동 모음 | "이 사이트 납품된 버전의 시험 이력 한 번에" 불가능 |

### 검사관 시각의 핵심 질문 (실패하면 안 되는 케이스)

1. *"이 PLC 인터락 수정은 누가 언제 시험했고, 결과는?"* — CR + 시험문서로 즉답
2. *"v2.1.0.0에 무슨 변경이 들어 있고 각각 시험 통과했나?"* — CR 묶음 + 각 시험 결과
3. *"이 장비(6040DA)는 KTL 인증된 버전을 깔고 있나?"* — 장비 현재버전 vs 인증버전 일치 확인
4. *"승인되지 않은 버전이 현장에 깔린 곳 없나?"* — 미승인 릴리스 + 배포 이력 교차

## 3. 타사 / 업계 표준

### 3.1 동급 보안검색 업체

| 회사 | 변경관리 방식 |
|------|---------------|
| **Smiths Detection** | 자체 PLM(Product Lifecycle Management) — Polarion 기반 추정. 모든 CR·요구사항·시험·릴리스가 ID로 연결 |
| **Rapiscan** | Jira + Confluence + 자체 빌드 시스템. 규제 대응(TSA, ECAC) 위해 별도 QMS 트랙 |
| **Nuctech** | 사내 ERP + 별도 품질 모듈. 인증 추적은 인증기관(CCC, ECAC)별 |
| **Leidos / IDSS** | DOORS(요구사항) + Jira(CR) + Vault(문서). 미국방·DHS 인증 대응 |

### 3.2 적용 가능한 산업 표준

- **ISO 9001** — 일반 품질경영 (변경관리 추적 필수)
- **ISO 13485** — 의료기기. 우리 RX6040MD가 의료용이면 해당
- **IEC 62304** — 의료기기 SW 라이프사이클 (CR ↔ 시험 ↔ 릴리스 추적 의무)
- **ECAC / TSA** — 항공보안 (인증 시점 SW 버전 잠금 + 변경 시 재인증)
- **KTL 보안검색 인증** — 한국. 인증 후 버전 업 시 변경요약·시험결과 제출

### 3.3 핵심 공통 원칙

> **추적성(Traceability) 매트릭스**
> 요구사항 → 변경요청 → 설계 → 코드 → 시험 → 릴리스 → 배포 → 인증
> 어느 단계에서든 위·아래 양방향 연결 가능해야 함

**우리 규모(SW팀)에서 풀스택 PLM은 과잉.** 핵심만 잘라서 가벼운 추적성 매트릭스만 구축하면 됨.

## 4. 권장 설계 — 4개 새 개념

### 4.1 ChangeRequest (변경요청) — 이번 핵심

폴더 `01.기능추가_Detector` 한 개 = ChangeRequest 한 건.

```
ChangeRequest
─────────────
- crNumber       : "CR-6040DA-001"  (모델별 일련번호 자동 부여)
- title          : "Detector 기능 추가"
- changeType     : 기능추가 / 기능개선 / 버그수정 / UI리뉴얼 / 시험
- component      : Detector / PLC / UI / AI / TIP / 통합 / 기타
- model          : AIXAC-RX6040DA  (필수, 다중 모델은 별도 CR)
- description    : 변경 사유와 상세
- status         : 요청 / 개발중 / 시험중 / 완료 / 보류 / 취소
- owner          : 담당자
- requestedAt    : 요청일
- completedAt    : 완료일
- sourceFolder   : "03.SW/01.제품/C.AIXAC-RX6040DA/01.기능추가_Detector"
                   ← 기존 폴더 경로를 그대로 보존 (참고용)
- releaseId      : Release FK  (이 CR이 포함된 릴리스 — N:1 또는 N:M)
```

### 4.2 ChangeRequestDocument (품질문서)

폴더 안의 .hwp/.docx/.xlsx/.pdf 파일들의 메타.

```
ChangeRequestDocument
─────────────────────
- changeRequestId : FK
- documentType    : 요구사항서 / 시험계획서 / 시험결과서 / 품질검사결과서 / 승인서 / 기타
- title           : "PLC 통신 시험결과서 v1.0"
- filePath        : "\\server\03.SW\01.제품\...\03.기능추가_UI\03.시험결과서.hwp"
                    또는 IntraWorks 업로드 경로
- issuedDate      : 작성/발행일
- approver        : 승인자
- revision        : 개정번호 ("1.0", "1.1", "2.0")
- isFinal         : 최종본 여부
```

### 4.3 Release ↔ ChangeRequest (다대다)

```
ReleaseChangeRequest (조인 테이블)
──────────────────────────────────
- releaseId
- changeRequestId
- includedAt   : 이 CR이 릴리스에 포함된 시점 (보통 release.buildDate)
```

→ **한 릴리스 = 여러 CR 묶음**.
→ 릴리스 상세 화면에 "이 버전에 포함된 변경" 자동 표시.
→ CR 상세 화면에 "어느 릴리스에 포함됐는지" 자동 표시.

### 4.4 Certification (인증 이력)

```
Certification
─────────────
- model           : AIXAC-RX6040DA
- certBody        : KTL / 항공보안인증 / ECAC / 기타
- certNumber      : "KTL-2026-X1234"
- certifiedVersion: Release FK  (인증 시점 버전 잠금)
- certifiedDate   : 인증일
- validUntil      : 만료일 (있는 경우)
- status          : 유효 / 만료 / 갱신중 / 취소
- documentPath    : 인증서 PDF 경로
- notes           : 비고
```

## 5. 워크플로우 — 검사관이 와도 즉답

```
[1] 새 기능 발의
    └─ /change-requests/new
         · 모델 선택, 유형, 제목, 설명
         · CR-{model}-{NNN} 자동 생성
         · sourceFolder 입력 (폴더 작업 병행 시)
         · 상태 = "요청"

[2] 개발 시작 → 상태 "개발중"

[3] 시험 진행 → 상태 "시험중"
    · 시험계획서·시험결과서 업로드 (Document)
    · 품질검사결과서 업로드

[4] 릴리스 만들기 → /releases/new
    · 버전 입력 (예: v2.1.0.0)
    · 포함된 CR들 체크박스로 선택 ← 핵심
    · 빌드 산출물 경로
    · changelog 자동 생성 (선택된 CR들의 title 모음)

[5] 현장 배포 → /deploys/new
    · 장비 + 버전
    · 자동으로 CR 추적 가능 (Release → CRs)

[6] 인증 받기 → /certifications/new
    · 모델 + 인증기관 + 인증버전(Release) + 만료일
    · 인증서 PDF 첨부

[7] 검사관 방문
    · 모델 클릭 → 인증 탭 → "v2.1.0.0이 KTL 인증됨"
    · 그 v2.1.0.0 클릭 → 포함된 CR 5건 보임
    · 각 CR 클릭 → 시험결과서·품질검사서·승인서 모두 다운로드 가능
    · 그 v2.1.0.0이 깔린 장비 4대도 보임
```

## 8. 명시적 비범위

다음은 이번 플랜에서 **하지 않음**:

- 풀스택 PLM(Polarion 등) 도입 — 과잉
- 요구사항 관리(DOORS 스타일) — CR 단위로 충분
- 코드 변경 추적 — git 영역, 따로 안 함
- 테스트 자동화 / CI 연계 — SW팀 별도 영역
- 외부 인증기관 시스템 연동 — 수동 입력만
- 고객사 사용 통계 — 별도 영역
- 의사결정 워크플로우(승인 단계, 결재) — 단순 status 필드로만

## 9. 결정 필요사항

진행 전 사용자 확인 부탁:

1. **CR 단위 정의**: 폴더 1개 = CR 1개로 보면 충분한가? (현재 6040DA 폴더에 14개 폴더 있음 = 14개 CR 마이그레이션)
2. **문서 저장 방식**: 사내 파일서버 경로만 vs 실제 업로드 (한글 파일은 보통 1~5MB)
3. **모델 단위 분리**: CR은 모델 1개에 종속? 또는 여러 모델 공통 CR 가능?
   - 현재 폴더 구조는 모델 단위 → 모델 1개 권장
   - 같은 기능을 여러 모델에 적용 시 CR도 모델별로 1개씩 생성
4. **인증 추적 우선순위**: KTL부터? 모든 인증기관 한 번에?
5. **기존 데이터 마이그레이션**: 6040DA 폴더 14개를 자동 import 할 스크립트 만들어야 하나? 아니면 사용자가 시간 날 때 수동 등록?

## 11. 한 줄 요약
